# 将想法转化为设计方案

## 概述

通过自然的协作对话，帮助将想法转化为完整的设计和规范。

首先理解当前项目上下文，然后逐个提问来精化想法。一旦理解了要构建什么，就分段呈现设计（每段 200-300 字），每段之后确认是否正确再继续。

## 流程

**理解想法：**
- 先了解当前项目状态（文件、文档、最近提交）
- 逐个提问来精化想法
- 优先使用选择题，开放式问题也可以
- 每条消息只问一个问题 - 如果某个话题需要深入探索，拆分成多个问题
- 聚焦于理解：目的、约束、成功标准

**代码搜索（理解项目时必须使用）：**

搜索优先级（按顺序尝试）：

1. **Repotalk MCP** - 跨仓库搜索
   - 适用：查找最佳实践、参考实现、跨项目模式
   - 示例：搜索字节内部类似功能的实现方式

2. **bcindex MCP** - 语义搜索
   - 适用：自然语言定位代码，术语不确定时
   - 示例："处理用户认证的代码"、"HTTP 中间件实现"

3. **byte-lsp MCP** - 符号定位
   - 适用：符号名明确时，查找定义、引用、实现
   - 工具：`go_to_definition`、`find_references`、`search_symbols`
   - 示例：查找 `IUserHandler` 接口的所有实现

4. **Glob/Grep/Read** - 兜底方案
   - 适用：MCP 不可用或需要精确文本匹配时

**探索方案：**
- 提出 2-3 种不同方案及其权衡
- 以对话方式呈现选项，给出你的推荐和理由
- 先说推荐方案，再解释原因

**呈现设计：**
- 一旦你认为理解了要构建什么，就开始呈现设计
- 分成 200-300 字的段落
- 每段之后询问是否正确
- 覆盖：架构、组件、数据流、错误处理、测试
- 如果有不清楚的地方，随时回溯澄清

## 设计完成后

**文档沉淀：**
- 将验证过的设计写入 `.bytecoding/plans/YYYY-MM-DD-<主题>-design.md`
- 将设计文档提交到 git

**继续实现（如果需要）：**
- 询问："准备好开始实现了吗？"
- 引导用户使用 `/bytecoding:plan` 生成任务列表

## 核心原则

- **一次一个问题** - 不要用多个问题轰炸用户
- **优先选择题** - 比开放式问题更容易回答
- **严格 YAGNI** - 从所有设计中移除不必要的功能
- **探索替代方案** - 在确定之前总是提出 2-3 种方案
- **增量验证** - 分段呈现设计，逐段确认
- **保持灵活** - 有不清楚的地方就回溯澄清
